# 小学数学结构化知识库建设规范（Spec）

## 1. 背景与目标

当前资料由“学生版/教师版”两套课程文档组成，课程编号体系基本一致，规模约 358 份（每版约 179 份），格式以 `.doc` 为主、少量 `.docx`。  
目标是构建可持续演进的“知识库底座”，同时满足：

- 面向教学内容的结构化沉淀（可检索、可复用、可追溯）
- 面向 AI 的标准化接口供给（RAG / Agent 调用）
- 面向前端应用的数据服务能力（课程浏览、知识图谱、练习推荐）

## 2. 设计原则

- 内容与展示解耦：知识模型独立于前端页面与 AI 提示词
- 原始可追溯：任何结构化字段都可回溯到源文档片段
- 版本可管理：知识对象、题目对象、讲解对象都支持版本号
- 增量可迭代：先建“最小可用底座”，再扩展推理与推荐能力
- 双版本协同：学生版与教师版关联到同一个“课程主题实体”
- 主从源清晰：教师版作为主知识源，学生版作为呈现与差异校验源

## 2.1 教师版与学生版的数据源策略

- 主源（Canonical）：教师版，承载题目答案、解析、教学提示等完整信息
- 从源（Projection）：学生版，承载无答案练习视图与排版参考
- 结构化后不再依赖“原始学生版是否有答案”，可通过字段级输出控制生成“学生视图”
- 学生版仍有价值：用于版式一致性、文本差异校验、练习打印模板生成
- 兜底规则：当教师版个别课程缺失或异常时，允许学生版补齐题干与章节骨架

## 3. 业务对象模型（核心实体）

### 3.1 Topic（知识主题）

- `topic_id`: 课程唯一标识（建议来自编号体系，如 `4-2-1`）
- `title`: 主题名
- `domain`: 领域（计算/代数/几何/数论/应用题/计数/逻辑）
- `grade_band`: 学段（低/中/高年级）
- `difficulty`: 难度等级（1-5）
- `tags`: 标签集合
- `prerequisites`: 前置主题列表
- `learning_objectives`: 学习目标列表
- `common_mistakes`: 常见错误列表

### 3.2 Concept（概念）

- `concept_id`, `name`, `definition`
- `core_formulas`: 公式与适用条件
- `solution_patterns`: 解题范式（模型法、方程法、图示法等）
- `counter_examples`: 易错反例

### 3.3 Problem（题目）

- `problem_id`
- `topic_id`
- `stem`: 题干
- `answer`: 标准答案
- `analysis_steps`: 结构化步骤
- `method_tags`: 方法标签
- `difficulty`: 难度
- `source_ref`: 来源文档引用

### 3.4 Explanation（讲解）

- `explanation_id`
- `audience`: `student` / `teacher`
- `summary`: 讲解摘要
- `key_points`: 关键点
- `teaching_tips`: 教学建议（教师版重点）
- `misconception_fix`: 易错纠偏说明

### 3.5 Asset（资源与追溯）

- `asset_id`, `file_path`, `edition`, `checksum`
- `extract_status`: 提取状态
- `chunk_refs`: 文本切片定位

## 4. 存储与分层架构

## 4.1 分层

- L0 原始层（Raw）：保留原始 `.doc/.docx`
- L1 清洗层（Normalized）：统一转为结构化中间文本（Markdown/JSON）
- L2 语义层（Curated）：Topic/Concept/Problem/Explanation 标准对象
- L3 服务层（Serving）：检索索引、向量索引、API 服务

### 4.2 数据存储建议

- 元数据与关系：PostgreSQL
- 文本检索：PostgreSQL Full-text 或 OpenSearch
- 语义检索：向量库（pgvector 或专用向量数据库）
- 对象文件：本地对象目录或对象存储

## 5. 文档处理流水线

### 5.1 Ingestion（导入）

- 扫描目录并建立清单（文件、版本、编号、路径、哈希）
- 将 `.doc` 统一转为 `.docx` 或可解析中间格式
- 文本抽取并段落化，保留章节层级

### 5.2 Structuring（结构化）

- 规则提取：标题、知识点、例题、练习、答案、解析
- 模型抽取：学习目标、方法标签、易错点、前置依赖
- 双版本对齐：同编号课程聚合为同一 Topic 下的两个 Audience 视图
- 字段投影：同一题目保留完整答案字段，但对学生端接口默认隐藏 `answer/analysis`

### 5.3 Quality（质检）

- 完整性校验：必填字段覆盖率
- 一致性校验：题干-答案-解析逻辑一致
- 可追溯校验：结构化对象均含 `source_ref`

## 6. 标准化接口设计（面向 AI 与前端）

### 6.1 查询接口（示例）

- `GET /topics`：主题列表（支持领域/难度/年级过滤）
- `GET /topics/{id}`：主题详情（含双版本讲解）
- `GET /problems`：题目检索（标签/难度/方法过滤）
- `GET /graph/{topic_id}`：知识依赖图谱

### 6.2 AI 调用接口（示例）

- `POST /ai/retrieve`：基于问题意图返回知识片段+题目示例
- `POST /ai/lesson-plan`：生成教案草稿（教师视角）
- `POST /ai/hint`：生成分层提示（学生视角）

### 6.3 返回规范

- 统一 Envelope：`code/message/data/trace_id`
- 支持 `source_ref` 回传，确保可解释性
- 支持版本参数：`knowledge_version`

## 7. 里程碑与范围控制

### M1（底座）

- 完成目录清单、格式统一、基础字段结构化
- 建立 Topic + Problem + Explanation 的最小模型
- 提供基础查询 API
- 建立“教师主源、学生投影”的出题与打印视图

### M2（增强）

- 加入知识依赖图谱、前置关系与错因标签
- 接入向量检索，支持混合检索
- 建立自动化质检报表

### M3（应用）

- 对接 AI Agent 调用接口
- 对接前端知识地图、课程页、练习页
- 引入内容版本发布与回滚机制

## 8. 风险与应对

- `.doc` 老格式解析不稳定：先做格式转换与抽样验收
- 命名与编号存在异常：建立命名规范映射表
- 结构化质量波动：规则+模型双通道并引入人工抽检
- AI 幻觉风险：强制 `source_ref` 回溯与置信度阈值

## 9. 验收标准（Spec 级）

- 至少 95% 文档完成可读提取
- 至少 90% 课程完成 Topic 级结构化对象
- 任一 API 返回结果均可回溯源文档
- 学生版与教师版可通过课程编号稳定关联

## 10. 增量扩展与持续接入

### 10.1 接入机制

- 新文档进入 `staging` 区后自动计算哈希并判重
- 命中已有哈希则跳过重抽取，仅更新来源引用关系
- 新增或变更文档触发增量流水线，仅重建受影响 Topic/Problem

### 10.2 版本策略

- 采用 `knowledge_version` 进行语义版本发布
- 每次发布保留变更清单（新增/修改/下线对象）
- 支持按版本回放，保障 AI 与前端可重复查询

### 10.3 可运营性

- 提供“补充文档接入模板”，约束标题、编号、章节层级
- 提供每日/每周增量质检报表，跟踪覆盖率与异常率
- 支持人工复核工作台，对低置信抽取结果二次确认
